CLI Snask

A interface de linha de comando snask é o ponto de entrada único para compilar, executar, gerenciar projetos, escanear bindings C, validar manifestos e diagnosticar a instalação. Esta referência cobre todos os subcomandos e flags disponíveis.

snask build · snask run · snask setup · snask init · snask doctor · snask add / remove · snask om scan · snask explain · snask snif · Perfis · Flags de tamanho

snask build

Compila um ou mais arquivos Snask. Pode ser usado no modo avulso (passando um arquivo fonte diretamente) ou no modo projeto (dentro de um diretório SPS com snask.snif).

Modo avulso

Compila um único arquivo sem necessidade de manifesto:

snask build main.snask --output app
./app

O argumento posicional é o caminho para o arquivo de entrada. A flag --output (ou -o) define o nome do binário gerado. Se omitida, o compilador usa o nome do arquivo sem extensão.

Modo projeto (SPS)

Dentro de um diretório que contém snask.snif, o comando sem argumentos lê as configurações do manifesto automaticamente: 

cd meu_projeto
snask build

O manifesto define a entrada principal (main), perfil de compilação, dependências e scripts. O comando respeita todas as seções do snask.snif e gera o lockfile snask.lock automaticamente.

Dica: use --release-size, --min-runtime, --tiny ou --extreme para controlar o nível de otimização de tamanho do binário. Consulte a seção Flags de tamanho para detalhes.

snask run

Compila e executa o projeto em um único passo. Equivalente a executar snask build && ./<output>. 

snask run

O nome do binário é resolvido a partir do manifesto snask.snif ou do nome do diretório. A saída é armazenada em um diretório temporário ou no diretório build/ do projeto.

Atenção: em projetos com perfil systems, certifique-se de que o ambiente tenha as permissões necessárias, pois o binário pode acessar hardware ou memória diretamente.

snask setup

Configura o ambiente de desenvolvimento: instala o plugin Neovim (snask.nvim), adiciona o diretório de binários ao PATH e verifica as dependências do sistema. 

snask setup

O comando realiza as seguintes etapas:

EtapaDescrição
Neovim pluginCopia snask.nvim para o diretório de plugins do Neovim
PATHAdiciona ~/.snask/bin ao ~/.bashrc ou ~/.zshrc
Verificação LLVMConfirma que o LLVM 18 está instalado e acessível
LSPGarante que snask-lsp está presente e funcional

snask init

Cria um novo projeto SPS (Snask Project System) com a estrutura de diretórios padrão e um manifesto snask.snif gerado automaticamente. 

snask init meu_app
cd meu_app
snask build

A estrutura gerada é:

meu_app/
  snask.snif        # Manifesto do projeto
  src/
    main.snask      # Ponto de entrada
  snask.lock        # Lockfile (gerado no primeiro build)

O comando aceita um único argumento: o nome do diretório do projeto. Se o diretório já existir, o comando falha a menos que a flag --force seja fornecida.

Nota: o manifesto gerado usa o perfil humane como padrão. Para projetos systems, edite o campo profile manualmente no snask.snif.

snask doctor

Diagnostica a instalação do Snask e do ecossistema. Verifica a presença do compilador, LSP, LLVM, Neovim (opcional) e dependências de sistema. 

snask doctor

Saída típica:

Snask 0.4.1-alpha
  Compilador:     /usr/bin/snask
  LSP:            /usr/bin/snask-lsp
  LLVM:           18.1.6
  Neovim:         0.10.0
  Dependências:   OK

Se alguma verificação falhar, o comando exibe uma mensagem descritiva e sugere ações corretivas. Ideal para usar após a instalação ou ao encontrar erros inesperados.

snask add / snask remove

Gerencia dependências do projeto adicionando ou removendo entradas na seção [dependencies] do snask.snif, com resolução de versões e atualização do lockfile. 

snask add logger
snask add json@1.0.0
snask remove logger
ComandoDescrição
snask add <pacote>Adiciona a dependência na versão mais recente compatível
snask add <pacote>@<versão>Adiciona com versão específica
snask add <pacote> --registry <url>Adiciona de um registry customizado
snask remove <pacote>Remove a dependência do manifesto e lockfile
Status: parcial. A resolução de dependências avançada (gráfico completo, resolução de conflitos) ainda está em maturação. Para projetos simples o funcionamento é estável.

snask om scan

Escaneia um cabeçalho C e gera bindings OM (Object Model) para uso no Snask. Essencial para interoperar com bibliotecas C existentes. 

snask om scan /usr/include/sdl2/SDL.h --lib SDL2 --output om/SDL.om
FlagDescrição
<header>Caminho para o arquivo de cabeçalho C a ser escaneado
--lib <nome>Nome da biblioteca (usado nos nomes dos módulos gerados)
--output <caminho>Diretório ou arquivo de saída para os bindings
--sysroot <dir>Diretório raiz do sistema (para cross-compilação)
--include <dir>Diretório adicional de includes

O scanner processa #define, typedef, struct, enum e protótipos de função, gerando bindings tipados e seguros.

snask explain

Exibe uma explicação detalhada para códigos de erro, diagnóstico ou aviso emitidos pelo compilador. 

snask explain S1002
snask explain T4001
snask explain W2005

Cada código segue o formato X####, onde a letra indica a categoria:

CódigoCategoriaExemplo
SSintaxeS1002 — token inesperado
TTipoT4001 — incompatibilidade de tipos
WAviso (warning)W2005 — variável não utilizada
MMemóriaM3001 — vazamento de zona detectado
LLinkingL1000 — símbolo não encontrado
IInterno do compiladorI0001 — erro no backend LLVM
Dica: consulte a página Diagnósticos para a lista completa de códigos e suas descrições.

snask snif

Subcomando dedicado ao formato SNIF (Snask Native Interchange Format). Inclui ferramentas de formatação, validação e geração de esquema.

snask snif fmt

Formata um arquivo SNIF, normalizando indentação, espaçamento e ordem dos campos:

snask snif fmt snask.snif
snask snif fmt snask.snif --write

Sem --write, a saída formatada é exibida no terminal. Com --write, o arquivo é sobrescrito in-place.

snask snif schema

Exibe o esquema atual do formato SNIF em diferentes formatos:

snask snif schema          # esquema textual
snask snif schema --md     # tabela Markdown

A saída em Markdown é útil para documentação ou para incluir em READMEs. O esquema lista todos os campos válidos, tipos esperados e se são obrigatórios.

snask snif check

Valida um arquivo SNIF contra o esquema oficial:

snask snif check snask.snif

O validador verifica tipos de campo, valores permitidos, integridade estrutural e consistência das referências. Retorna código 0 se válido, ou lista os erros encontrados.

SubcomandoDescrição
snif fmt <arquivo> [--write]Formata arquivo SNIF
snif schema [--md]Exibe o esquema SNIF
snif check <arquivo>Valida arquivo contra o esquema

Perfis de compilação

O Snask oferece dois perfis de compilação que controlam o nível de abstração, segurança de memória e acesso a hardware. O perfil é definido via flag --profile ou no manifesto snask.snif.

PerfilFlagPadrãoDescrição
Humane --profile humane Sim Gerenciamento automático de memória via zonas. Sem ponteiros crus. Foco em produtividade e segurança. Ideal para aplicações, scripts e ferramentas do dia a dia.
Systems --profile systems Não Acesso a ponteiros, alocação manual, operações de bits e hardware. Exige blocos @unsafe. Ideal para kernels, drivers, emuladores e sistemas embarcados. 
# Humane (padrão)
snask build main.snask --output app

# Systems
snask build main.snask --profile systems --output app
Convivência: os dois perfis podem coexistir no mesmo programa. Você começa no perfil Humane e ativa Systems apenas nos módulos que precisam de acesso de baixo nível.

Flags de otimização de tamanho

Flags que controlam o nível de otimização voltado à redução do tamanho do binário gerado. Podem ser combinadas com o perfil de compilação.

FlagDescriçãoUso típico
--release-size Build otimizado para tamanho. Ativa passes do LLVM focados em Oz. Redução geral de tamanho sem sacrificar funcionalidade.
--min-runtime Remove partes opcionais do runtime (coletor de zonas, diagnóstico de memória, debug symbols). Binários mínimos para distribuição ou embedded.
--tiny Modo mais agressivo que combina --release-size e --min-runtime com stripping total de símbolos. Projetos com restrições severas de espaço.
--extreme Experimento de tamanho mínimo. Remove ainda mais componentes do runtime e aplica transformações experimentais no LLVM. Pesquisa e exploração de limites. Pode quebrar funcionalidades. 
snask build main.snask --output app --release-size
snask build main.snask --output app --tiny
snask build main.snask --output app --extreme
Cuidado: as flags --tiny e --extreme podem remover funcionalidades do runtime que seu programa utiliza. Teste exaustivamente antes de distribuir o binário.

Flags globais

FlagDescrição
--helpExibe ajuda do comando ou subcomando

--verboseSaída detalhada com informações de depuração

Códigos de saída

CódigoSignificado
0Sucesso
1Erro genérico (falha de compilação, validação, etc.)
2Erro de uso (flag inválida, argumento faltando)
3Erro de instalação (LLVM ausente, LSP não encontrado)
4Erro de configuração (manifesto inválido, lockfile corrompido)
