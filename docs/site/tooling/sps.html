SPS e SNIF

SPS (Snask Project System) é o sistema de gerenciamento de projetos do Snask. SNIF (Snask Native Interchange Format) é o formato de manifesto que define metadados, dependências, build e scripts do projeto. Juntos, formam a base para compilações reproduzíveis e gerenciamento de dependências.

Criar um projeto

Use o comando snask init para gerar a estrutura inicial de um projeto SPS: 

snask init meu_app
cd meu_app

A estrutura gerada contém:

meu_app/
  snask.snif        # Manifesto do projeto (formato SNIF)
  src/
    main.snask      # Código fonte principal
  snask.lock        # Lockfile (gerado no primeiro build)

O manifesto é criado com valores padrão: perfil humane, entrada src/main.snask e versão 0.1.0. Edite o arquivo para ajustar nome, versão, dependências e configurações de build.

Manifesto snask.snif

O arquivo snask.snif é o manifesto do projeto. Escrito em SNIF, ele define todas as configurações do projeto em seções organizadas.

Exemplo mínimo

[project]
name = "hello_snask"
version = "0.4.1-alpha"
main = "src/main.snask"

Seção [project]

CampoTipoObrigatórioDescrição
namestringSimNome do projeto
versionstringSimVersão semântica (ex: 0.4.1-alpha)
mainstringSimCaminho relativo para o arquivo de entrada
descriptionstringNãoDescrição curta do projeto
authorsarray de stringNãoLista de autores
licensestringNãoIdentificador SPDX da licença

Exemplo completo

[project]
name = "hello_snask"
version = "0.4.1-alpha"
main = "src/main.snask"
description = "Um projeto de exemplo"
authors = ["Alice", "Bob"]
license = "MIT"

[dependencies]
logger = "1.0.0"
json = "0.2.0"

[build]
profile = "humane"
target = "native"
strip = false

[scripts]
test = "snask build && ./test_runner"
bench = "snask build --profile systems && ./bench"
doc = "snask snif schema --md > SCHEMA.md"

Seção [dependencies]

Lista as dependências externas do projeto. Cada chave é o nome do pacote e o valor é a restrição de versão.

FormatoExemploSignificado
Versão exatalogger = "1.0.0"Apenas a versão 1.0.0
Caret (compatível)logger = "^1.0.0"Qualquer versão >=1.0.0 e <2.0.0
Tilde (aproximada)logger = "~1.0.0"Qualquer versão >=1.0.0 e <1.1.0
Wildcardlogger = "*"Qualquer versão (não recomendado para produção)
Intervalologger = ">=1.0.0 <2.0.0"Intervalo arbitrário de versões
 
[dependencies]
logger = "1.0.0"
json = "^0.2.0"
sdl = "~2.28.0"

O Snask resolve as versões no momento do build (ou ao executar snask add) e registra as versões exatas no lockfile.

Seção [build]

Configurações de compilação do projeto. Substitui flags de linha de comando quando definidas no manifesto.

CampoTipoPadrãoDescrição
profilestring"humane"Perfil de compilação: humane ou systems
targetstring"native"Triplet da arquitetura alvo (ex: x86_64-linux-gnu)
stripboolfalseRemove símbolos de depuração do binário
opt_levelint2Nível de otimização (0-3, ou z para tamanho)
release_sizeboolfalseEquivalente a --release-size
min_runtimeboolfalseEquivalente a --min-runtime
outputstring—Nome do binário de saída
 
[build]
profile = "humane"
target = "native"
strip = false
opt_level = 2

Seção [scripts]

Define comandos personalizados que podem ser executados via snask run-script <nome>. Útil para automação de tarefas como teste, benchmark e documentação. 

[scripts]
test = "snask build && ./test_runner"
bench = "snask build --profile systems && ./bench"
doc = "snask snif schema --md > SCHEMA.md"
lint = "snask snif check snask.snif"
ChaveComando associado
testsnask run-script test
benchsnask run-script bench
docsnask run-script doc
lintsnask run-script lint

Formato SNIF alternativo (histórico)

Antes da sintaxe de seções ([project], [dependencies]), o SNIF era escrito em um formato de objeto único. Ainda reconhecido pelo compilador para compatibilidade retroativa: 

{
  package: {
    name: "store",
    version: "0.1.0",
    entry: "main.snask"
  },
  dependencies: {
    json: "*"
  },
  build: {
    profile: "humane",
    opt_level: 2
  },
  scripts: {
    run: "snask run"
  }
}
MapeamentoFormato modernoFormato histórico
[project]name, version, mainpackage.name, package.version, package.entry
[dependencies]dependencies (seção)dependencies (objeto)
[build]build (seção)build (objeto)
[scripts]scripts (seção)scripts (objeto)
Nota: o formato de seções é recomendado para projetos novos. O formato de objeto único é mantido para não quebrar projetos existentes.

Lockfile snask.lock

O arquivo snask.lock é gerado automaticamente no primeiro build e atualizado sempre que as dependências mudam. Ele registra as versões exatas resolvidas e hashes de integridade, garantindo compilações reproduzíveis. 

# snask.lock
version = 1

[[packages]]
name = "logger"
version = "1.0.0"
sha256 = "a1b2c3d4e5f6..."
registry = "https://registry.snask.dev"

[[packages]]
name = "json"
version = "0.2.0"
sha256 = "f6e5d4c3b2a1..."
registry = "https://registry.snask.dev"
CampoDescrição
versionVersão do formato do lockfile
[[packages]]Lista de pacotes resolvidos
nameNome do pacote
versionVersão exata resolvida
sha256Hash SHA-256 do conteúdo do pacote (verificação de integridade)
registryURL do registry de origem (opcional, usado para validação)

O lockfile deve ser versionado no repositório (commitar no git) para que todos os desenvolvedores usem as mesmas versões das dependências.

Importante: nunca edite o snask.lock manualmente. Use snask add e snask remove para gerenciar dependências — o lockfile é atualizado automaticamente.

Comandos relacionados

ComandoDescrição

# Fluxo de trabalho típico
snask init meu_app
cd meu_app
snask add logger
snask build
snask run

Formato SNIF — referência completa

SNIF (Snask Native Interchange Format) é um formato de serialização de dados usado pelo manifesto snask.snif e pelo lockfile. Abaixo estão todos os elementos da sintaxe.

Tipos de valor

TipoExemploDescrição
string"hello"Texto entre aspas duplas
integer42Número inteiro (64 bits)
float3.14Número de ponto flutuante
booltrueBooleano (true / false)
nullnullAusência de valor

Objetos

Delimitados por chaves {}, campos separados por vírgula:

{
  name: "exemplo",
  versao: 2,
  ativo: true
}

Arrays

Delimitados por colchetes [], elementos separados por vírgula:

[1, 2, 3, "quatro"]

Comma-final (trailing comma)

Vírgula após o último elemento é opcional e permitida — válida:

{
  a: 1,
  b: 2,
}

[1, 2, 3,]

Literais tipados

SNIF suporta literais com prefixo de tipo para valores especiais:

LiteralExemploDescrição
@date@date("2024-12-01")Data ISO 8601
@dec@dec("3.14159265358979323846")Decimal de precisão arbitrária
@enum@enum("debug")Valor enumerado
@path@path("src/main.snask")Caminho de arquivo normalizado
@regex@regex("^\\w+$")Expressão regular

Big integers

Inteiros além do limite de 64 bits podem ser escritos com o sufixo big ou usando @dec:

valor_muito_grande: 12345678901234567890big
preciso: @dec("12345678901234567890.123456789")

Referências

SNIF suporta referências a valores definidos em outro ponto do mesmo documento:

SintaxeSignificado
&nomeDefine um âncora chamada nome no valor atual
*nomeReferencia o valor da âncora nome
 
{
  config_padrao: &padrao {
    timeout: 30,
    retry: 3
  },
  servidor_a: {
    url: "http://a.com",
    config: *padrao
  },
  servidor_b: {
    url: "http://b.com",
    config: *padrao
  }
}

O que NÃO é válido em SNIF

InválidoExemploMotivo
Barewords sem aspasname: helloStrings devem estar entre aspas duplas: "hello"
Separador =name = "hello"SNIF usa : como separador, não =. O formato de seções ([project]) é uma exceção que usa =.
Vírgula faltando{a: 1 b: 2}Campos em objetos devem ser separados por vírgula
Comentários de barra// comentárioSNIF não suporta comentários (o manifesto usa seções e o lockfile usa # como exceção)
Tabulação em indentação\tname: "x"SNIF usa espaços para indentação (tabs não são permitidos)
Dica: use snask snif check para validar seu arquivo SNIF contra o esquema oficial e snask snif fmt para formatá-lo automaticamente.

Status atual

O sistema SPS e o formato SNIF estão em estágio parcial. São funcionais e úteis para projetos simples, mas algumas funcionalidades ainda estão em maturação:

FuncionalidadeStatusObservação
Projeto simples (sem dependências)EstávelBuild e run funcionam sem ressalvas
Dependências básicasEstávelResolução de versão única e lockfile funcionam
Workspaces (múltiplos pacotes)ParcialEstrutura básica existe, resolução entre pacotes ainda limitada
Resolução avançada de dependênciasParcialGráfico completo e resolução de conflitos ainda em desenvolvimento
Registries remotosParcialRegistry público em fase de protótipo
Cache de dependênciasExperimentalCache local funcional, mas sem garbage collection
Formato SNIF (moderno)EstávelSintaxe de seções e validação completas
Formato SNIF (histórico)EstávelCompatibilidade retroativa mantida
Atenção: para projetos que exigem resolução complexa de dependências ou workspaces com muitos pacotes, monitore o arquivo docs/reference/FEATURE_STATUS.md para atualizações sobre o progresso dessas funcionalidades.
