Capítulo 5

Gerenciamento de Memória (OM)

O Snask introduz o OM-Snask-System, uma alternativa revolucionária ao Garbage Collector (GC) e ao gerenciamento manual (malloc/free).

O Problema do GC

Linguagens como Java e Python usam um Garbage Collector que "pausa" seu programa para limpar a memória. Em sistemas de alta performance ou emuladores, essas pausas causam quedas de frames (stuttering). O Snask resolve isso com Zonas.

O Conceito de Zonas

Uma Zona é um bloco de tempo e espaço onde a memória vive. Quando a zona termina, toda a memória alocada dentro dela é liberada instantaneamente, sem busca recursiva e sem pausa. 

class main {
    fun start() {
        zone "processamento" {
            let dados = carregar_arquivo_gigante()
            // Os dados vivem aqui...
        } 
        // AO CHEGAR AQUI, 'dados' e toda a memória usada na zona 
        // são liberados imediatamente pelo hardware/runtime.
    }
}

Zonas de Frame

Em jogos ou emuladores, o Snask permite criar zonas que duram exatamente 1 frame (1/60 de segundo). Isso garante que seu jogo nunca consuma mais memória do que o necessário para um único ciclo. 

while true {
    zone "frame" {
        atualizar_logica()
        renderizar_pixels()
    } // Limpeza total do frame anterior aqui!
}

New Stack vs New Heap

No Snask, você tem controle sobre onde seus dados são colocados:

  • Stack: Rápido, automático, para dados pequenos e locais.
  • Arena/Zone: Para grupos de dados que devem morrer juntos.
  • Heap: Para dados persistentes (raramente necessário no estilo idiomático Snask).
