Perfil Systems

O perfil systems é a porta de entrada para código de baixo nível em Snask. Ele libera primitivas determinísticas, acesso a memória crua e operações de bit/registrador que o perfil Humane oculta por segurança.

Visão Geral

Snask oferece dois perfis de compilação: Humane (padrão) e Systems. O perfil Humane prioriza produtividade, segurança e gerenciamento automático de memória via zonas OM. O perfil Systems existe para quando você precisa de controle total sobre a máquina.

No perfil Systems, você pode:

  • Acessar primitivas de memória crua: mem_alloc, mem_free, mem_copy;
  • Usar blocos @unsafe para operações que o analisador não consegue verificar;
  • Trabalhar com ponteiros (ptr) e aritmética de endereços;
  • Chamar funções built-in de bit/registrador sem restrições;
  • Controlar layout de memória e endianness manualmente.
Importante: Systems não é "Humane sem segurança". É um perfil diferente com regras diferentes. O código Systems deve ser pequeno, testado e cercado por APIs Snask seguras.

Quando Usar Systems

O perfil Systems é a escolha certa para domínios onde o custo do gerenciamento automático supera seus benefícios. Os casos de uso típicos incluem:

EmuladoresMapeamento direto de registradores e memória da CPU alvoNES, Game Boy, Chip-8
Parsers bináriosLeitura/escrita de bytes em offsets específicosELF, PNG, TCP/IP

Se seu código se encaixa em um desses domínios, o perfil Systems oferece as ferramentas certas. Caso contrário, comece com Humane — você sempre pode migrar módulos específicos para Systems conforme a necessidade.

Build

Para compilar um projeto com o perfil Systems, use a flag --profile systems:

snask build apps/nes_emulator/nes_master.snask --profile systems --output /tmp/snask_nes

O compilador irá:

  1. Habilitar as primitivas de baixo nível (memória crua, ponteiros, builtins de bit);
  2. Permitir blocos @unsafe sem warning;
  3. Não inserir runtime OM para zonas (você gerencia manualmente);
  4. Emitir chamadas LLVM otimizadas para o alvo.
Atenção: nem todo código Snask compila no perfil Systems. Código que depende de zonas, recursos automáticos ou coleta de lixo do runtime precisa do perfil Humane. Módulos Systems devem ser intencionalmente isolados.

Memória Crua

O perfil Systems expõe funções de alocação direta. Todo acesso a memória crua deve ocorrer dentro de blocos @unsafe: 

class main {
    fun start() {
        @unsafe {
            // Aloca 64KB de memória zero-inicializada
            let mem: ptr = mem_alloc_zero(65536)

            // Escreve o reset vector do 6502 em 0xFFFC
            mem_write_u8(mem, 0xFFFC, 0x00)
            mem_write_u8(mem, 0xFFFD, 0x80)

            // Lê o vetor como little-endian u16
            let pc: u16 = mem_read_u16(mem, 0xFFFC)

            // Libera a memória alocada
            mem_free(mem)
        }
    }
}

Perigo: chamar mem_free duas vezes no mesmo ponteiro (double-free) ou acessar memória após liberá-la (use-after-free) causa comportamento indefinido. O runtime não protege contra esses erros no perfil Systems.

Primitivas Úteis

O perfil Systems disponibiliza um conjunto de funções built-in para manipulação de tipos, bits e operações aritméticas com detecção de overflow. Essas funções são essenciais para implementar emuladores, processadores e protocolos binários.

Conversão

Conversões explícitas entre tipos numéricos sem perda de sinal implícita:

let valor: u16 = 0xABCD
let baixo: u8 = as_u8(valor)      // 0xCD
let alto: u8 = hi_u8(valor)       // 0xAB
let inteiro: i32 = as_i32(valor)  // 0x0000ABCD
let offset: usize = as_usize(inteiro)

Bits e Flags

Manipulação direta de bits para flags de CPU e registro de estado:

let flags: u8 = 0b00000000

// Testa se o bit 1 (carry) está setado
let carry: bool = bit_test(flags, 1)

// Seta o bit 3 (decimal mode no 6502)
let flags: u8 = bit_set(flags, 3)

// Escreve um valor booleano em um bit específico
let flags: u8 = flag_write(flags, 1, true)  // seta carry

// Limpa um bit
let flags: u8 = bit_clear(flags, 3)

6502 — Operações de 16 bits

Funções específicas para trabalhar com registradores de 16 bits comuns em CPUs clássicas:

let endereco: u16 = 0xABCD
let lo: u8 = lo_u8(endereco)   // 0xCD — byte baixo
let hi: u8 = hi_u8(endereco)   // 0xAB — byte alto
let reconstruido: u16 = make_u16(hi, lo)  // 0xABCD

Overflow Aritmético

Operações que preservam carry/borrow/overflow — fundamentais para emulação de CPU:

// Adição com carry (útil para ADC no 6502)
let (result: u8, carry: bool) = carry_add_u8(0x80, 0x80)

// Subtração com borrow (útil para SBC no 6502)
let (result: u8, borrow: bool) = borrow_sub_u8(0x00, 0x01)

// Adição com detecção de overflow signed (útil para V flag)
let (result: i8, overflow: bool) = overflow_add_i8(0x40, 0x40)

// Subtração com detecção de overflow signed
let (result: i8, overflow: bool) = overflow_sub_i8(0x7F, 0xFF)

Modelo de Memória

O perfil Systems oferece três regimes de memória. Entender quando usar cada um é essencial para escrever código eficiente e correto.

Stack

A pilha é o regime padrão para variáveis locais. Toda variável declarada dentro de uma função vive na stack e é destruída quando a função retorna. Não há custo de alocação — é apenas um ajuste de ponteiro. 

fun compute() {
    let a: u8 = 10       // stack
    let b: u8 = 20       // stack
    let c: u8 = a + b    // stack
} // tudo destruído ao sair

Heap (alocação manual)

Dados que precisam sobreviver à função que os criou devem ser alocados no heap. No perfil Systems, isso é feito manualmente com mem_alloc / mem_free: 

@unsafe {
    let buffer: ptr = mem_alloc(4096)
    // ... usa o buffer ...
    mem_free(buffer)
}

Memória raw (mapeada)

Para emuladores e kernels, é comum mapear memória em endereços fixos ou simular um espaço de endereçamento. O perfil Systems permite alocar blocos grandes e acessá-los por offset, exatamente como uma CPU real faria: 

@unsafe {
    // 64KB — espaço de endereçamento do 6502
    let mem: ptr = mem_alloc_zero(65536)

    // Mapeia cartucho no início
    mem_copy(mem, cartucho, 0x8000)

    // Stack pointer começa em 0x01FF (cresce para baixo)
    mem_write_u8(mem, 0x01FD, 0x00)

    // CPU começa a execução no reset vector
    let pc: u16 = mem_read_u16(mem, 0xFFFC)
}

Segurança no Perfil Systems

O perfil Systems não elimina segurança — ele a redefine. Em vez de proteger o programador do erro humano, ele exige que o programador prove que o código está correto. As regras são:

Isolamento de @unsafe

Todo código que acessa memória crua, manipula ponteiros ou chama operações não verificáveis deve estar dentro de um bloco @unsafe. Esse isolamento permite que revisores identifiquem rapidamente as regiões críticas.

Módulos pequenos e verificáveis

O código Systems deve ser concentrado em módulos pequenos com responsabilidade única. Um emulador de NES, por exemplo, pode ter um módulo cpu.snask com ~300 linhas de código Systems, enquanto o resto do projeto (UI, áudio, entrada) fica em Humane.

APIs seguras como barreira

Módulos Systems devem exportar APIs seguras (sem @unsafe) para o resto do projeto. O código Humane nunca precisa ver ponteiros ou chamar mem_free: 

// cpu.snask — módulo Systems
class NesCpu {
    // API pública — segura, sem ponteiros expostos
    fun step() -> u8 { @unsafe { ... } }
    fun read_byte(address: u16) -> u8 { @unsafe { ... } }
    fun write_byte(address: u16, value: u8) { @unsafe { ... } }
}

Dica: você pode usar os dois perfis no mesmo projeto. Módulos críticos compilam com Systems; o resto fica em Humane. Apenas isole bem as fronteiras com APIs seguras.

Boas Práticas para Misturar Humane e Systems

Isole em módulos

Cada módulo Systems deve ter um propósito único e claro. Ex: cpu.snask, apu.snask, ppu.snask. Não misture lógica de negócio com código de baixo nível no mesmo arquivo.

Exporte APIs seguras

Módulos Systems devem expor apenas funções seguras (sem @unsafe na assinatura). O chamador Humane nunca deve precisar entrar em @unsafe.

Evite @unsafe vazado

Nunca retorne ponteiros ou recursos não gerenciados para código Humane. Use tipos opacos ou IDs numéricos para referenciar recursos internos.

Teste os dois lados

Testes de unidade para o módulo Systems (com @unsafe) e testes de integração para o código Humane que consome as APIs seguras.

Documente contratos

Cada função Systems deve documentar: pré-condições (ex: ponteiro não nulo, tamanho mínimo), pós-condições e responsabilidade de liberação.

Mantenha o @unsafe curto

Um bloco @unsafe ideal tem 3-10 linhas. Se precisar de mais, extraia funções auxiliares. Blocos longos são difíceis de revisar.
