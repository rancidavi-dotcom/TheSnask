Memória Orquestrada

OM-Snask-System

O OM (Orquestrador de Memória) é o sistema único de gerenciamento de ciclo de vida, recursos nativos e interop com C da linguagem Snask. Em vez de você escrever o contrato manualmente, o Snask deduz o contrato a partir da sua intenção.

Frase guia: em vez de você escrever malloc, free, destroy e ownership manual, o Snask deduz o ciclo de vida a partir da estrutura do código.

Visão Geral

O OM resolve três problemas que linguagens tradicionais tratam separadamente:

  • Memória — quem aloca, quem libera, quando.
  • Recursos externos — arquivos, handles, conexões, texturas GPU.
  • Interoperabilidade C — chamar bibliotecas nativas sem vazar ponteiros crus para o usuário.

A ideia central é unificar esses três mundos sob o mesmo sistema de zonas: escopos delimitados onde toda alocação e todo recurso morrem ao sair.

O OM opera em dois perfis: no perfil Humane, tudo é gerenciado automaticamente por zonas. No perfil Systems, você acessa as primitivas de baixo nível (mem_alloc, mem_free, ponteiros) através de blocos @unsafe.

A hierarquia de memória do OM organiza os dados em camadas:

CamadaUsoVidaObservação

stackobjetos curtosframe atualideal para helpers temporários

Atenção: o sistema completo de análise formal (escape analysis, borrow checking, zone depth) ainda está em desenvolvimento. O que existe hoje é funcional para a maioria dos casos, mas o ferramental de verificação estática está sendo implementado.

Módulo 1: Zonas

Uma zona é um escopo nominado que agrupa alocações e recursos. Tudo que nasce dentro de uma zona morre quando a zona fecha.

Teoria

Zonas são a unidade fundamental do OM. Elas funcionam como "escopos com nome": você declara uma zona com zone "nome" { ... } e todo valor alocado dentro dela — seja no heap do runtime, seja um recurso externo — tem seu ciclo de vida atrelado ao fechamento da zona.

Isso elimina a necessidade de rastrear manualmente quando liberar cada alocação. O padrão é conhecido como RAII regional (Regional Resource Acquisition Is Initialization): a região (zona) é dona de tudo que contém.

Cada zona tem três responsabilidades:

  • agrupar alocações temporárias;
  • delimitar a vida dos dados;
  • limpar recursos registrados ao sair.

Sintaxe

zone "frame" {
    let buffer = String::new(1024)
    let dados = read_file("config.json")
    processar(dados, buffer)
} // buffer e dados são liberados aqui

O mesmo conceito vale para recursos C — ao sair da zona, o runtime chama os destrutores associados aos recursos automaticamente: 

zone "window" {
    let window = sdl2.create_window(...)
    let renderer = sdl2.create_renderer(window, -1, 0)
    // Ao sair, SDL_DestroyWindow e SDL_DestroyRenderer são chamados
}

Aninhamento

Zonas podem ser aninhadas. A zona interna tem acesso aos recursos da externa, mas o inverso não é verdadeiro — ao fechar a zona interna, apenas os recursos dela são liberados.

zone "request" {
    let req = parse_http(request)
    zone "response" {
        let resp = build_response(req)
        send(resp)
    } // resp liberado
    // req ainda vivo aqui
} // req liberado
Dica: uma zona deve viver o menor tempo possível — quanto mais cedo fechar, menos memória retida. Se você precisa que um valor sobreviva à zona atual, use promote para promovê-lo ao heap.

Promoção (heap)

Quando um valor precisa viver além da zona, use promote para movê-lo ao heap persistente: 

zone "load" {
    let temp = carregar_dados()
    let persistente = promote(temp)
} // temp é liberado, persistente continua vivo

Boas práticas

  • Nomeie zonas com propósito semântico: "frame", "request", "batch".
  • Uma zona deve viver o menor tempo possível — quanto mais cedo fechar, menos memória retida.
  • Evite zonas muito profundas (mais de 3 níveis) — prefira extrair funções.
  • Cada função pode ter sua própria zona. Não force tudo em uma zona gigante.

Módulo 2: Arenas

Uma arena é um alocador linear de curta duração. Ideal para processamento em lote e operações temporárias.

Teoria

Enquanto zonas são sobre escopo, arenas são sobre eficiência. Uma arena pré-aloca um bloco contíguo de memória e serve alocações incrementais sem passar pelo alocador global do sistema. Quando a arena é destruída, toda a memória é liberada de uma vez — sem passar item por item.

Os benefícios da arena:

  • alocação O(1) — apenas incrementa um ponteiro;
  • boa localidade de cache — dados ficam contíguos;
  • limpeza em massa — um único free libera tudo;
  • menos fragmentação — sem buracos entre alocações;
  • menos chamadas a malloc/free do sistema.

Isso é particularmente útil em:

  • Processamento de quadros em jogos (frame arena).
  • Parsers e compiladores (alocações temporárias durante análise).
  • Requisições HTTP (alocar headers, bodies, depois liberar tudo).

Sintaxe

// Perfil Humane — arena via zona
zone "frame" {
    // O runtime decide se usa arena internamente
    let particles = Array::new(1000)
    atualizar(particles)
} // liberação em massa

// Perfil Systems — arena explícita
@unsafe {
    let arena = mem_arena_create(1024 * 64) // 64KB
    let a: ptr = mem_arena_alloc(arena, 256)
    let b: ptr = mem_arena_alloc(arena, 512)
    mem_arena_destroy(arena) // ambos liberados
}

Comparação

CaracterísticaZonaArena
PropósitoCiclo de vidaPerformance
LiberaçãoPor recurso (destrutores)Em massa (reset)
AninhamentoSimNão (sequencial)
OverheadMédio (tracking individual)Baixo (bump allocator)

Dica: no perfil Humane, use zonas. O runtime pode otimizar zonas curtas para usar uma arena internamente de forma transparente.

Módulo 3: Recursos Externos

Handles de runtime, ponteiros C e qualquer recurso com destrutor conhecido pode ser registrado no OM.

Teoria

Nem tudo é memória. Arquivos abertos, conexões de rede, texturas GPU, handles de janela — tudo isso são recursos que precisam ser liberados. O OM permite registrar destrutores personalizados para qualquer tipo de recurso, e o sistema de zonas garante que o destrutor seja chamado no fechamento da zona.

Registro de recurso

Quando você chama uma função C que retorna um recurso, o runtime do OM registra automaticamente o ponteiro e seu destrutor associado: 

zone "render" {
    let texture = load_texture("hero.png")
    // O runtime sabe como destruir texture
    // porque o contrato C declarou o destructor
    draw(texture)
} // texture destruída automaticamente

Recursos nativos do runtime

O runtime do Snask já registra destrutores para:

  • Arquivos — read_file, write_file
  • Janelas GUI — gui_window, snaskgui_window
  • Conexões HTTP — http_get, http_post
  • Memória systems — mem_alloc, mem_alloc_zero

Funções internas do runtime

O runtime OM expõe funções internas que são usadas pelo codegen, não pelo usuário diretamente: 

s_zone_register(ptr, destructor, type_name)
s_om_resource_ptr(resource_handle, expected_type)

A primeira registra um recurso na zona atual. A segunda extrai o ponteiro bruto de um handle para fazer a chamada C. O usuário nunca chama essas funções — elas são o contrato interno entre o codegen LLVM e o runtime.

No perfil Humane, o uso dessas funções dentro de uma zona é seguro e o recurso é liberado automaticamente. No perfil Systems, você precisa estar dentro de @unsafe para chamar mem_free manualmente.

Módulo 4: Contratos C e o Scanner

O scanner analisa headers C e deduz contratos de ciclo de vida automaticamente. Quando a heurística falha, você corrige com .om.snif.

O problema

Bibliotecas C tradicionais exigem que o programador gerencie manualmente:

  • quem aloca (malloc, SDL_CreateWindow, fopen);
  • quem libera (free, SDL_DestroyWindow, fclose);
  • ponteiros nulos;
  • ownership: emprestado ou transferido?;
  • ordem de destruição.

O OM-Snask-System automatiza isso. Em vez de bindings escritos à mão, o scanner lê o header C diretamente com Clang e produz um contrato em memória que o compilador usa para gerar chamadas LLVM nativas.

Fluxo do scanner

Header C
    → Clang parse
    → extração de funções, tipos, enums, macros
    → heurísticas de ownership
    → contrato OM em memória
    → patch .om.snif opcional
    → contrato final usado no LLVM

O que o scanner extrai

  • funções e assinaturas C
  • parâmetros e tipos de retorno
  • enums e macros numéricas → constantes Snask
  • tipos opacos (structs forward-declared)
  • candidatos a construtor (funções que retornam Type*)
  • candidatos a destrutor (funções void destroy(Type*))

O contrato gerado

O scanner produz um contrato como este (em memória ou exportável): 

library sdl2

constant INIT_VIDEO: 32
constant WINDOW_HIDDEN: 8
constant QUIT: 256

resource SDLWindow:
    c_type: SDL_Window*
    constructor: SDL_CreateWindow
    destructor: SDL_DestroyWindow
    surface_type: sdl2.SDLWindow
    safety: SAFE
    reason: constructor `SDL_CreateWindow` returns `SDL_Window*` and paired destructor `SDL_DestroyWindow` accepts `SDL_Window*`

function create_window:
    c_function: SDL_CreateWindow
    surface: sdl2.create_window
    input: value
    output: resource
    c_return_type: SDL_Window*
    c_param_types: char*, int, int, int, int, Uint32
    safety: SAFE
    reason: constructor `SDL_CreateWindow` is registered as OM resource `sdl2.SDLWindow`

Níveis de exposição

O contrato classifica cada função em um nível de segurança:

NívelSignificadoExemplo
SAFEPode ser chamada diretamente pela superfície Snasksdl2.delay(50)
COPY_ONLYPode ser usada, mas o OM copia o resultado para memória Snasksdl2.get_platform() (copia string)
BLOCKEDNão é exposta — função perigosa ou não mapeadadestrutores, callbacks, void*

Regras de segurança do scanner

O scanner é conservador por design. Ele prefere bloquear uma função útil a expor uma função insegura:

Constantes

Macros numéricas e enums viram constantes no namespace:

sdl2.INIT_VIDEO
sdl2.WINDOW_HIDDEN
sdl2.QUIT

Para SDL2, o prefixo SDL_ é removido automaticamente.

Funções simples

Funções com parâmetros numéricos, strings e retornos simples são expostas diretamente:

stdio.puts("texto")
sdl2.delay(50)
sdl2.init(sdl2.INIT_VIDEO)

Recursos opacos

O scanner procura pares construtor/destrutor:

SDL_Window* SDL_CreateWindow(...) → sdl2.create_window(...)
void SDL_DestroyWindow(SDL_Window*) → escondido atrás do OM

Métodos de recurso

Se uma função recebe o recurso como primeiro argumento, ela vira chamada segura:

sdl2.render_clear(renderer)
sdl2.render_present(renderer)

Mapeamento de nomes

CSnask
SDL_Initsdl2.init
SDL_CreateWindowsdl2.create_window
SDL_RenderClearsdl2.render_clear
SDL_WINDOW_HIDDENsdl2.WINDOW_HIDDEN

O que o scanner NÃO consegue fazer hoje

Essas limitações são conhecidas e definem onde o scanner deve ser conservador:

  • callbacks C e function pointers
  • variadic functions complexas
  • structs C por valor
  • arrays e buffers mutáveis
  • ownership ambíguo (ex: ponteiro de saída que pode ser free pelo caller)
  • recursos com destrutores dependentes de contexto
  • APIs que exigem ordem global complexa
  • ponteiros globais
  • unions e bitfields
  • macros que expandem para expressões complexas

Para todas essas situações, a saída é o patch .om.snif.

Nota: o OM-Snask-System não gera uma cópia literal da API C. Uma API Snask deve expor intenção, não ponteiros. O sistema esconde SDL_Window*, SDL_DestroyWindow etc. quando isso pode ser provado com segurança.

Módulo 5: Tutorial .om.snif — Quando o Scanner Falha

O scanner automático cobre 80% dos casos. Para os 20% restantes — nomes ruins, ownership não convencional, funções perigosas — você escreve um patch .om.snif. Este tutorial ensina tudo do zero.

Filosofia: o .om.snif não é um binding gigante. Ele é um patch opcional sobre o contrato deduzido. O que não estiver no patch continua vindo do scanner. Uma biblioteca enorme pode ter apenas um arquivo pequeno de exceções.

5.1 Quando o scanner falha

O scanner é bom para bibliotecas que seguem convenções C comuns (create/free, open/close, new/destroy). Ele falha quando:

  • a biblioteca usa nomes de função não convencionais;
  • o destrutor não segue o padrão void destroy(Type*);
  • uma função perigosa foi classificada como segura;
  • uma função segura foi classificada como BLOCKED;
  • o ownership é ambíguo e precisa de declaração explícita;
  • a função usa tipos que o scanner não mapeia automaticamente.

5.2 Diagnóstico: entendendo o erro

Quando o scanner não consegue mapear uma função, ele produz erros como:

OM cannot map C parameter type `SDL_Rect*` yet.
OM import blocked `sdl2.destroy_window`: cleanup/destructor functions
are hidden behind OM zone cleanup.
OM function `sdl2.render_clear` expects 1 C arguments, got 0.

Para diagnosticar o contrato gerado, use o comando om scan: 

snask om scan SDL2/SDL.h --lib sdl2 --output /tmp/sdl2.generated.om.snif

Isso gera o contrato completo que o scanner deduziu. Revise o arquivo para entender o que foi mapeado corretamente e o que foi bloqueado.

Use SNASK_OM_TRACE=1 para ver trace detalhado do runtime: 

SNASK_OM_TRACE=1 SDL_VIDEODRIVER=dummy ./meu_app

5.3 Anatomia de um arquivo .om.snif

O .om.snif usa a sintaxe SNIF (formato nativo do Snask) e organiza-se em seções: 

// contracts/minhalib.om.snif
library minhalib

// ── Corrigir nome de função ──
function open_context:
    c_function: ml_open_context
    surface: minhalib.open
    input: str
    output: resource
    safety: SAFE
    reason: library uses non-standard constructor name

// ── Bloquear função perigosa ──
function dangerous_op:
    c_function: ml_dangerous_op
    safety: BLOCKED
    reason: performs raw memory write without bounds checking

// ── Declarar recurso manualmente ──
resource MLContext:
    c_type: ML_Context*
    constructor: ml_open_context
    destructor: ml_close_context
    surface_type: minhalib.MLContext
    safety: SAFE
    reason: manual resource declaration for non-standard naming

// ── Declarar política COPY_ONLY ──
function get_version:
    c_function: ml_get_version
    surface: minhalib.version
    input: none
    output: copy
    safety: COPY_ONLY
    reason: returns const char* that must be copied to Snask string

5.4 Caso 1: Nome de função mal mapeado

Problema: a biblioteca chama a função ml_open_context, mas o scanner gerou minhalib.open_context (correto). Porém, o scanner não reconheceu que ela é um construtor porque o nome não segue o padrão create/new/open.

Sintoma: a função aparece como SAFE mas retorna um ponteiro sem destrutor associado.

Solução:

// contracts/minhalib.om.snif
resource MLContext:
    c_type: ML_Context*
    constructor: ml_open_context
    destructor: ml_close_context
    surface_type: minhalib.MLContext
    safety: SAFE
    reason: non-standard constructor name, manual pairing required

5.5 Caso 2: Função perigosa classificada como SAFE

Problema: a função ml_write_raw recebe void* e um tamanho, mas o scanner não consegue verificar se o buffer tem o tamanho correto.

Solução: bloquear explicitamente:

// contracts/minhalib.om.snif
function write_raw:
    c_function: ml_write_raw
    surface: minhalib.write_raw
    safety: BLOCKED
    reason: void* parameter cannot be verified for bounds safety

A função não aparecerá na superfície Snask. Se o usuário precisar dela, terá que usar @unsafe com um wrapper manual.

5.6 Caso 3: Função segura bloqueada pelo scanner conservador

Problema: o scanner bloqueou ml_get_status porque ela retorna const char* e o scanner não tem regra COPY_ONLY para esta biblioteca.

Solução: declarar como COPY_ONLY:

// contracts/minhalib.om.snif
function get_status:
    c_function: ml_get_status
    surface: minhalib.status
    input: none
    output: copy
    safety: COPY_ONLY
    reason: returns const char*, OM must copy to Snask string

5.7 Caso 4: Destrutor não convencional

Problema: a biblioteca usa ml_cleanup_context em vez de ml_destroy_context ou ml_close_context. O scanner não consegue parear com o construtor.

Solução:

// contracts/minhalib.om.snif
resource MLContext:
    c_type: ML_Context*
    constructor: ml_create_context
    destructor: ml_cleanup_context
    surface_type: minhalib.MLContext
    safety: SAFE
    reason: non-standard destructor name, explicitly paired

5.8 Caso 5: Ownership ambíguo

Problema: a função ml_borrow_buffer retorna um ponteiro para um buffer interno. O caller não é dono do buffer — não deve chamar free nem ter o recurso registrado em zona.

Solução:

// contracts/minhalib.om.snif
function borrow_buffer:
    c_function: ml_borrow_buffer
    surface: minhalib.borrow_buffer
    output: copy
    safety: COPY_ONLY
    reason: returns borrowed pointer, caller must NOT free; OM copies data

5.9 Estratégia: por onde começar

  1. Importe o header com import_c_om "minhalib.h" as minhalib
  2. Tente usar funções simples — as que compilarem estão mapeadas corretamente.
  3. Gere o contrato com snask om scan e revise as funções bloqueadas.
  4. Crie o patch contracts/minhalib.om.snif apenas para as exceções.
  5. Teste compilando e executando com SNASK_OM_TRACE=1.
  6. Repita até que todas as funções necessárias estejam expostas corretamente.

5.10 Exemplo completo: SDL2 com patch

Cenário: você quer usar SDL2 no Snask, mas o scanner bloqueou algumas funções que você precisa.

Passo 1: gere o contrato para diagnóstico:

snask om scan SDL2/SDL.h --lib sdl2 --output /tmp/sdl2.contrato.snif

Passo 2: revise o arquivo gerado. Identifique funções bloqueadas.

Passo 3: crie contracts/sdl2.om.snif:

// contracts/sdl2.om.snif
library sdl2

// SDL_GetPerformanceCounter retorna Uint64 — seguro, mas pode estar
// bloqueado se o scanner não reconheceu o tipo de retorno
function get_performance_counter:
    c_function: SDL_GetPerformanceCounter
    surface: sdl2.performance_counter
    input: none
    output: value
    safety: SAFE
    reason: returns Uint64, safe numeric type

// SDL_ShowSimpleMessageBox é seguro mas usa const char* que o scanner
// pode não ter mapeado para esta biblioteca
function show_simple_message_box:
    c_function: SDL_ShowSimpleMessageBox
    surface: sdl2.show_message_box
    input: value, str, str, resource
    output: value
    safety: SAFE
    reason: UI function with simple string parameters

// Garantir que SDL_GetError seja COPY_ONLY
function get_error:
    c_function: SDL_GetError
    surface: sdl2.error
    input: none
    output: copy
    safety: COPY_ONLY
    reason: returns const char*, must be copied to Snask string

Passo 4: compile e teste:

snask build meu_sdl2_app.snask --output app
SNASK_OM_TRACE=1 ./app

5.11 Boas práticas

Use .om.snif para:

  • renomear uma função mal mapeada;
  • adicionar safety: BLOCKED a uma função perigosa;
  • declarar uma regra especial de recurso (construtor/destrutor não convencional);
  • corrigir um construtor/destrutor não pareado pelo scanner;
  • documentar uma decisão de ownership.

Evite .om.snif para:

  • copiar centenas de assinaturas do header — deixe o scanner fazer isso;
  • expor ponteiros crus por conveniência — use COPY_ONLY em vez de SAFE com ponteiro;
  • recriar a API C inteira — o scanner já fez 80% do trabalho;
  • fazer bypass da segurança — se o scanner bloqueou, há motivo.

Mantenha os patches pequenos. Um .om.snif com 5-10 regras é saudável. Um .om.snif com 500 linhas é sinal de que o scanner precisa de melhoria.

5.12 Verificação do contrato final

Para ver o contrato final (scanner + patch aplicado), use: 

snask om scan SDL2/SDL.h --lib sdl2 --patches contracts/

Isso mostra o contrato consolidado. Compare com o contrato sem patch para confirmar que suas correções foram aplicadas.

Módulo 6: Borrow Checking e Escape Analysis

Análise estática em desenvolvimento para garantir segurança de memória em tempo de compilação.

Estado atual

O OM atual oferece gerenciamento automático de memória via zonas no runtime, mas a verificação estática completa ainda está em desenvolvimento:

  • estável — Zonas com liberação automática no runtime.
  • parcial — Escape analysis para detectar referências que saem da zona.
  • experimental — Borrow checker com regiões.

Exemplo de escape

O escape analysis (parcial) já consegue detectar casos simples:

fun f() {
    zone "tmp" {
        let local = String::new("dados")
        // ERROR (futuro): retornar referência que escapa da zona
        // return &local
    }
}

Garantias pretendidas

O sistema deve caminhar para estas garantias:

  • Código Snask seguro não recebe ponteiro C cru com ownership manual.
  • Destrutores reconhecidos não aparecem como funções públicas comuns.
  • Recursos opacos são limpos pela zona.
  • Funções não provadas como seguras são bloqueadas.
  • .om.snif é patch auditável, não binding manual gigante.
  • Chamadas C são emitidas como chamadas nativas no LLVM.
  • O usuário escreve Snask, não C.
Nota: por enquanto, o OM depende do runtime para liberar memória. O borrow checker estático será adicionado incrementalmente. No perfil Systems, use @unsafe para operações que o analisador ainda não consegue verificar.

Práticas Recomendadas

🔹 Prefira zonas a liberação manual

Mesmo no perfil Systems, use zonas sempre que possível. A liberação manual (mem_free) só deve ser usada quando você precisa de controle fino sobre o momento exato da desalocação.

🔹 Zonas curtas e nomeadas

Uma zona deve viver o mínimo necessário. Nomeie com propósito semântico para documentar a intenção. Ex: "frame", "batch", "request".

🔹 Evite zonas deep

Mais de 3 níveis de aninhamento de zonas é um sinal de que você deveria extrair uma função. Cada função pode ter sua própria zona.

🔹 Use @unsafe com disciplina

Isole operações @unsafe em funções pequenas e verificáveis. Nunca espalhe @unsafe pelo código — concentre em módulos systems bem definidos.

🔹 Contratos C explícitos

Sempre verifique os contratos gerados automaticamente pelo scanner C. Use arquivos .om.snif para corrigir quando necessário. Um contrato errado pode causar double-free ou leak.

🔹 Diagnostique com om scan

Use snask om scan para inspecionar o contrato gerado antes de compilar. Isso evita surpresas e ajuda a identificar rapidamente o que precisa de patch.

🔹 Teste com leak detector

Use snask test --leak-check para detectar vazamentos. O runtime do OM pode instrumentar todas as alocações em modo debug.

🔹 Trace com SNASK_OM_TRACE

Ative SNASK_OM_TRACE=1 para ver todo o ciclo de vida dos recursos: registro, uso e destruição. Essential para debug de contratos.

Critérios para promover o status do OM

Para sair de experimental para parcial ou estável, o OM-Snask-System precisa:

  • testes automatizados cobrindo stdio, zlib, sqlite e SDL2;
  • documentação de contrato .om.snif;
  • regras formais para safety (SAFE, COPY_ONLY, BLOCKED);
  • mensagens de erro consistentes;
  • suporte melhor a structs;
  • story clara para callbacks;
  • validação de linkagem multiplataforma;
  • fuzzing de headers ou corpus de libs C;
  • testes de cleanup em caminhos de erro;
  • auditoria do runtime OM.

Notas

O OM-Snask-System é um dos pilares da linguagem. Diferente de abordagens como Rust (ownership + borrow checker global) ou Zig (alocador explícito), o OM busca um meio-termo pragmático: você começa com zonas automáticas no perfil Humane e gradualmente ganha controle conforme necessário no perfil Systems.

O princípio final do OM: 

import_c_om "alguma_lib_c.h" as lib

class main
    fun start()
        zone "work":
            let resource = lib.create_resource()
            lib.use_resource(resource)
    // resource liberado automaticamente

Em vez de C manual: 

Resource *resource = create_resource();
if (!resource) return 1;
use_resource(resource);
destroy_resource(resource);

Para contribuir ou relatar problemas, abra uma issue no repositório oficial. Use SNASK_OM_TRACE=1 para incluir trace do runtime nos relatos.
