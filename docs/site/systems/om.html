Memória Orquestrada

OM-Snask-System

O OM (Orquestrador de Memória) é o sistema único de gerenciamento de ciclo de vida, recursos nativos e interop com C da linguagem Snask. Em vez de você escrever o contrato manualmente, o Snask deduz o contrato a partir da sua intenção.

Frase guia: em vez de você escrever malloc, free, destroy e ownership manual, o Snask deduz o ciclo de vida a partir da estrutura do código.

Visão Geral

Linguagens tradicionais forçam você a escolher entre dois extremos: produtividade com Garbage Collector (lento, imprevisível) ou controle manual de memória (rápido, perigoso, verboso). O OM oferece um terceiro caminho:

  • Zonas — escopos delimitados onde toda alocação morre ao sair.
  • Arenas — alocações temporárias em lote com liberação em massa.
  • Recursos externos — handles do runtime ou C com destrutores registrados.
  • Contratos C — bindings gerados automaticamente com anotações de ciclo de vida.

O OM opera em dois perfis: no perfil Humane, tudo é gerenciado automaticamente por zonas. No perfil Systems, você acessa as primitivas de baixo nível (mem_alloc, mem_free, ponteiros) através de blocos @unsafe.

Atenção: o sistema completo de análise formal (escape analysis, borrow checking, zone depth) ainda está em desenvolvimento. O que existe hoje é funcional para a maioria dos casos, mas o ferramental de verificação estática está sendo implementado.

Módulo 1: Zonas

Uma zona é um escopo nominado que agrupa alocações e recursos. Tudo que nasce dentro de uma zona morre quando a zona fecha.

Teoria

Zonas são a unidade fundamental do OM. Elas funcionam como "escopos com nome": você declara uma zona com zone "nome" { ... } e todo valor alocado dentro dela — seja no heap do runtime, seja um recurso externo — tem seu ciclo de vida atrelado ao fechamento da zona.

Isso elimina a necessidade de rastrear manualmente quando liberar cada alocação. O padrão é conhecido como RAII regional (Regional Resource Acquisition Is Initialization): a região (zona) é dona de tudo que contém.

Sintaxe

zone "frame" {
    let buffer = String::new(1024)
    let dados = read_file("config.json")
    processar(dados, buffer)
} // buffer e dados são liberados aqui

Aninhamento

Zonas podem ser aninhadas. A zona interna tem acesso aos recursos da externa, mas o inverso não é verdadeiro — ao fechar a zona interna, apenas os recursos dela são liberados.

zone "request" {
    let req = parse_http(request)
    zone "response" {
        let resp = build_response(req)
        send(resp)
    } // resp liberado
    // req ainda vivo aqui
} // req liberado

Boas práticas

  • Nomeie zonas com propósito semântico: "frame", "request", "batch".
  • Uma zona deve viver o menor tempo possível — quanto mais cedo fechar, menos memória retida.
  • Evite zonas muito profundas (mais de 3 níveis) — prefira extrair funções.

Módulo 2: Arenas

Uma arena é um alocador linear de curta duração. Ideal para processamento em lote e operações temporárias.

Teoria

Enquanto zonas são sobre escopo, arenas são sobre eficiência. Uma arena pré-aloca um bloco contíguo de memória e serve alocações incrementais sem passar pelo alocador global do sistema. Quando a arena é destruída, toda a memória é liberada de uma vez — sem passar item por item.

Isso é particularmente útil em:

  • Processamento de quadros em jogos (frame arena).
  • Parsers e compiladores (alocações temporárias durante análise).
  • Requisições HTTP (alocar headers, bodies, depois liberar tudo).

Sintaxe

// Perfil Humane — arena via zona
zone "frame" {
    // O runtime decide se usa arena internamente
    let particles = Array::new(1000)
    atualizar(particles)
} // liberação em massa

// Perfil Systems — arena explícita
@unsafe {
    let arena = mem_arena_create(1024 * 64) // 64KB
    let a: ptr = mem_arena_alloc(arena, 256)
    let b: ptr = mem_arena_alloc(arena, 512)
    mem_arena_destroy(arena) // ambos liberados
}

Módulo 3: Recursos Externos

Handles de runtime, ponteiros C e qualquer recurso com destrutor conhecido pode ser registrado no OM.

Teoria

Nem tudo é memória. Arquivos abertos, conexões de rede, texturas GPU, handles de janela — tudo isso são recursos que precisam ser liberados. O OM permite registrar destrutores personalizados para qualquer tipo de recurso, e o sistema de zonas garante que o destrutor seja chamado no fechamento da zona.

Registro de recurso

// Registrar um destrutor para um handle C
zone "render" {
    let texture = load_texture("hero.png")
    // O runtime sabe como destruir texture
    // porque o contrato C declarou o destructor
    draw(texture)
} // texture destruída automaticamente

Recursos nativos do runtime

O runtime do Snask já registra destrutores para:

  • Arquivos — read_file, write_file
  • Janelas GUI — gui_window, snaskgui_window
  • Conexões HTTP — http_get, http_post
  • Memória systems — mem_alloc, mem_alloc_zero

No perfil Humane, o uso dessas funções dentro de uma zona é seguro e o recurso é liberado automaticamente. No perfil Systems, você precisa estar dentro de @unsafe para chamar mem_free manualmente.

Módulo 4: Contratos C

Bindings para bibliotecas C com ciclo de vida deduzido automaticamente por scanner, corrigido manualmente via .om.snif.

Teoria

O OM inclui um scanner que analisa headers C e gera bindings Snask automaticamente. Para cada função que retorna um ponteiro ou recebe um recurso, o scanner tenta deduzir o contrato de ciclo de vida:

  • Ownership transfer — quem é dono do recurso após a chamada?
  • Borrow — o recurso é emprestado temporariamente?
  • Destructor — qual função deve ser chamada para liberar?

Quando o scanner não consegue deduzir (ou a dedução está errada), você pode corrigir com um arquivo .om.snif ao lado do header.

Exemplo de contrato

// Header C: sdl.h
// SDL_Surface* SDL_LoadBMP(const char* file)
// void SDL_FreeSurface(SDL_Surface* surface)

// Gerado automaticamente:
fun sdl_load_bmp(file: str) -> ptr @om(owns, destructor="sdl_free_surface")

// Uso em Snask:
zone "game" {
    let surf = sdl_load_bmp("hero.bmp")
    // surf será liberado com sdl_free_surface ao sair da zona
}

Arquivo .om.snif

// sdl.om.snif — correção manual do contrato
@contract "SDL_LoadBMP" {
    owns: return
    destructor: "SDL_FreeSurface"
    nullable: true
}
@contract "SDL_CreateRenderer" {
    owns: return
    destructor: "SDL_DestroyRenderer"
}
@contract "SDL_RenderPresent" {
    owns: none // não retorna recurso
}

Módulo 5: Borrow Checking e Escape Analysis

Análise estática em desenvolvimento para garantir segurança de memória em tempo de compilação.

Estado atual

O OM atual oferece gerenciamento automático de memória via zonas no runtime, mas a verificação estática completa ainda está em desenvolvimento:

  • estável — Zonas com liberação automática no runtime.
  • parcial — Escape analysis para detectar referências que saem da zona.
  • experimental — Borrow checker com regiões.

Exemplo de escape

O escape analysis (parcial) já consegue detectar casos simples:

fun f() {
    zone "tmp" {
        let local = String::new("dados")
        // ERROR (futuro): retornar referência que escapa da zona
        // return &local
    }
}
Nota: por enquanto, o OM depende do runtime para liberar memória. O borrow checker estático será adicionado incrementalmente. No perfil Systems, use @unsafe para operações que o analisador ainda não consegue verificar.

Práticas Recomendadas

🔹 Prefira zonas a liberação manual

Mesmo no perfil Systems, use zonas sempre que possível. A liberação manual (mem_free) só deve ser usada quando você precisa de controle fino sobre o momento exato da desalocação.

🔹 Zonas curtas e nomeadas

Uma zona deve viver o mínimo necessário. Nomeie com propósito semântico para documentar a intenção. Ex: "frame", "batch", "request".

🔹 Evite zones deep

Mais de 3 níveis de aninhamento de zonas é um sinal de que você deveria extrair uma função. Cada função pode ter sua própria zona.

🔹 Use @unsafe com disciplina

Isole operações @unsafe em funções pequenas e verificáveis. Nunca espalhe @unsafe pelo código — concentre em módulos systems bem definidos.

🔹 Contratos C explícitos

Sempre verifique os contratos gerados automaticamente pelo scanner C. Use arquivos .om.snif para corrigir quando necessário. Um contrato errado pode causar double-free ou leak.

🔹 Teste com leak detector

Use snask test --leak-check para detectar vazamentos. O runtime do OM pode instrumentar todas as alocações em modo debug.

Notas

O OM-Snask-System é um dos pilares da linguagem. Diferente de abordagens como Rust (ownership + borrow checker global) ou Zig (alocador explícito), o OM busca um meio-termo pragmático: você começa com zonas automáticas no perfil Humane e gradualmente ganha controle conforme necessário no perfil Systems.
