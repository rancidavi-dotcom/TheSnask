Low-Level em Snask

Snask foi projetada para ser confortável para iniciantes e ainda oferecer controle de baixo nível quando o projeto pede emuladores, runtimes, parsers binários ou kernels futuros. Esta página detalha o que significa "low-level" em Snask e como usar essas ferramentas com segurança.

O que é Low-Level em Snask

"Low-level" em Snask não é um modo binário (ligado/desligado). É um espectro controlado por três mecanismos que se complementam:

MecanismoO que ofereceQuando usar
Perfil SystemsPrimitivas de memória, ponteiros, builtins de bitO módulo inteiro precisa de controle fino
Blocos @unsafeAcesso a operações não verificáveisApenas algumas funções precisam de poder extra
OM (Orquestrador de Memória)Zonas, recursos, contratos CGerenciamento automático com interop C

Você pode usar os três no mesmo projeto. Um emulador de NES, por exemplo, usa o perfil Systems para o núcleo da CPU, @unsafe para acessar a memória mapeada do cartucho e OM para gerenciar recursos SDL2 (janela, áudio, input).

Frase guia: baixo nível em Snask é sobre controle, não sobre sofrimento. Você escolhe exatamente quanto controle precisa e o resto fica gerenciado.

Motivação: Emulador NES

O principal caso de uso que motivou o suporte low-level em Snask foi a emulação do NES (Nintendo Entertainment System). Um emulador de NES precisa de:

  • Registradores de 8 e 16 bits com operações aritméticas exatas;
  • Flags de CPU (carry, zero, interrupt, decimal, overflow, negative);
  • Memória de 64KB acessada por offset com endianness little-endian;
  • Determinismo total — cada instrução deve produzir o mesmo resultado sempre;
  • Sem pausas de GC ou alocação imprevisível durante a execução da CPU.

Snask foi construída para atender a esses requisitos sem sacrificar a segurança nas partes do código que não precisam de baixo nível.

Operações de Memória em Detalhe

O perfil Systems expõe um conjunto completo de operações de memória. Todas exigem @unsafe.

Alocação e Liberação

@unsafe {
    let a: ptr = mem_alloc(256)       // 256 bytes não inicializados
    let b: ptr = mem_alloc_zero(512)  // 512 bytes zerados
    mem_free(a)
    mem_free(b)
}

Cópia e Preenchimento

@unsafe {
    let src: ptr = mem_alloc(1024)
    let dst: ptr = mem_alloc(1024)

    mem_copy(dst, src, 1024)    // copia 1024 bytes de src para dst
    mem_fill_u8(dst, 0xFF, 64)  // preenche 64 bytes com 0xFF

    mem_free(src)
    mem_free(dst)
}

Acesso por Offset

Todas as funções de leitura/escrita usam offset a partir da base do ponteiro:

@unsafe {
    let mem: ptr = mem_alloc_zero(65536)

    // Escrita
    mem_write_u8(mem,  0x0000, 0xA9)       // LDA #imm
    mem_write_u8(mem,  0x0001, 0x42)       // imm = 0x42
    mem_write_u16(mem, 0xFFFC, 0x0000)     // reset vector = $0000

    // Leitura
    let opcode: u8   = mem_read_u8(mem, 0x0000)
    let operand: u8  = mem_read_u8(mem, 0x0001)
    let reset_pc: u16 = mem_read_u16(mem, 0xFFFC)

    mem_free(mem)
}

Aritmética de Ponteiros

Para acesso não linear, use ptr_add para deslocar o ponteiro:

@unsafe {
    let base: ptr = mem_alloc(4096)
    let deslocado: ptr = ptr_add(base, 2048)  // aponta para o meio do bloco
    mem_write_u8(deslocado, 0, 0xAA)
    mem_free(base)
}
FunçãoAssinaturaDescrição
mem_alloc(size: usize) -> ptrAloca sem inicializar
mem_alloc_zero(size: usize) -> ptrAloca e zera
mem_free(ptr)Libera memória
mem_copy(dst: ptr, src: ptr, size: usize)Copia bytes
mem_fill_u8(ptr, value: u8, size: usize)Preenche com byte
mem_read_u8(ptr, offset: usize) -> u8Lê byte no offset
mem_write_u8(ptr, offset: usize, value: u8)Escreve byte no offset
mem_read_u16(ptr, offset: usize) -> u16Lê u16 little-endian
mem_write_u16(ptr, offset: usize, value: u16)Escreve u16 little-endian
ptr_add(ptr, offset: usize) -> ptrAvança ponteiro

Endianness

Snask adota little-endian como padrão para todas as operações multi-byte de memória. Isso significa que:

  • mem_read_u16(ptr, 0) lê ptr[0] como byte baixo e ptr[1] como byte alto;
  • mem_write_u16(ptr, 0, 0xABCD) escreve 0xCD em ptr[0] e 0xAB em ptr[1];
  • O processador 6502 e a maioria dos sistemas modernos (x86, ARM) usam little-endian, então não há reversão de bytes.
@unsafe {
    let mem: ptr = mem_alloc(4)
    mem_write_u16(mem, 0, 0xABCD)
    let lo: u8 = mem_read_u8(mem, 0)   // 0xCD
    let hi: u8 = mem_read_u8(mem, 1)   // 0xAB
    mem_free(mem)
}
Nota: se você precisar de big-endian (rede, alguns formatos de arquivo), terá que fazer a reversão manualmente com lo_u8 / hi_u8 / make_u16. Suporte nativo a byte swapping está nos planos.

Regras do @unsafe

O bloco @unsafe é o mecanismo que delimita onde o programador assume responsabilidade manual. As regras são claras:

RegraExplicação
Isolamento totalNenhuma operação @unsafe pode estar fora de um bloco @unsafe declarado.
Curto e diretoUm bloco @unsafe deve conter apenas as operações críticas. Ideal: 3-10 linhas.
Sem vazamentoPonteiros crus não podem ser retornados para código seguro. Use tipos opacos ou índices.
Pré-condições documentadasTodo @unsafe deve ter um comentário explicando por que é seguro naquele contexto.
Testado obrigatoriamenteCada bloco @unsafe deve ter teste de unidade cobrindo a operação.
Riscos: dentro de @unsafe, o compilador não protege contra buffer overflow, use-after-free, double-free, ponteiro nulo ou violação de aliasing. Tudo é responsabilidade do programador.

Exemplo: CPU 6502 Simplificada

Abaixo, um exemplo de como implementar o ciclo de步 (step) de uma CPU 6502 usando as primitivas low-level de Snask. A CPU tem registradores A, X, Y, PC (16 bits), SP (8 bits) e P (flags). 

class Cpu6502 {
    // Registradores
    var a: u8     // Acumulador
    var x: u8     // Índice X
    var y: u8     // Índice Y
    var pc: u16   // Program Counter
    var sp: u8    // Stack Pointer (0x0100 + sp)
    var p: u8     // Status Flags (NV-BDIZC)
    var mem: ptr  // Ponteiro para 64KB de memória

    fun init() {
        @unsafe {
            self.mem = mem_alloc_zero(65536)
        }
        self.pc = 0x0000
        self.sp = 0xFD
        self.p  = 0b00100000  // Bit 5 sempre setado
    }

    fun destroy() {
        @unsafe {
            mem_free(self.mem)
        }
    }

    // Busca um byte da memória e incrementa PC
    fun fetch_byte() -> u8 {
        @unsafe {
            let opcode: u8 = mem_read_u8(self.mem, as_usize(self.pc))
        }
        self.pc = self.pc + 1
        opcode
    }

    // Lê um operando de 16 bits (little-endian)
    fun fetch_word() -> u16 {
        @unsafe {
            let word: u16 = mem_read_u16(self.mem, as_usize(self.pc))
        }
        self.pc = self.pc + 2
        word
    }

    // Executa uma instrução
    fun step() {
        let opcode: u8 = self.fetch_byte()

        if opcode == 0xA9 {  // LDA #imm
            let value: u8 = self.fetch_byte()
            self.a = value
            self.update_nz(value)
        }
        else if opcode == 0x8D {  // STA abs
            let addr: u16 = self.fetch_word()
            @unsafe {
                mem_write_u8(self.mem, as_usize(addr), self.a)
            }
        }
        else if opcode == 0xAD {  // LDA abs
            let addr: u16 = self.fetch_word()
            @unsafe {
                self.a = mem_read_u8(self.mem, as_usize(addr))
            }
            self.update_nz(self.a)
        }
        else if opcode == 0x69 {  // ADC #imm
            let value: u8 = self.fetch_byte()
            let carry: bool = bit_test(self.p, 0)
            let (result, new_carry) = carry_add_u8(self.a, value)
            if carry {
                let (result2, _) = carry_add_u8(result, 1)
                result = result2
            }
            self.a = result
            self.p = flag_write(self.p, 0, new_carry)
            self.update_nz(self.a)
            // Overflow (V) seria calculado aqui
        }
        else if opcode == 0x00 {  // BRK
            // Interrupção — implementação omitida
        }
    }

    fun update_nz(value: u8) {
        self.p = flag_write(self.p, 1, value == 0)      // Z
        self.p = flag_write(self.p, 7, bit_test(value, 7))  // N
    }
}

Observe que os blocos @unsafe são pequenos e focados: apenas as operações de leitura/escrita de memória. A lógica de controle (opcodes, flags) fica em código Snask seguro.

Padrão: isole o @unsafe em funções auxiliares (fetch_byte, fetch_word) para que o resto do código não precise lidar com ponteiros. Isso também facilita testes — você pode mockar a memória.

Boas Práticas para Código Low-Level

Pré-aloque tudo

Para código determinístico (emuladores, áudio), aloque toda a memória necessária na inicialização. Evite mem_alloc durante a execução principal.

Use índices, não ponteiros

Em vez de passar ponteiros entre funções, passe offsets ou índices. Isso reduz o risco de ponteiros soltos e facilita a depuração.

Uma responsabilidade por @unsafe

Cada bloco @unsafe deve fazer exatamente uma operação: ler, escrever ou alocar. Nunca misture lógica de negócio com operações não seguras.

Teste com endereços limítrofes

Sempre teste acesso a memória nos limites: offset 0, offset final, offset logo após o final. Um erro de off-by-one é o bug mais comum em código low-level.

Mantenha o estado explícito

Registradores de CPU, flags, contadores — tudo deve ser variável explícita. Nunca confie em estado global implícito ou efeitos colaterais escondidos.

Documente o modelo de memória

Para cada bloco de memória alocado, documente: tamanho, layout, ownership (quem libera), e se é acessado concorrentemente.

Resumo do Modelo

  • humane é o perfil padrão para produtividade e runtime completo.
  • systems libera primitivas determinísticas e memória crua controlada.
  • @unsafe delimita trechos onde o programador assume responsabilidade manual.
  • OM continua existindo para recursos, zonas e limpeza determinística.

Código de baixo nível deve ser pequeno, testado e cercado por API Snask segura. O objetivo não é transformar todo programa em memória manual; é permitir que a parte que precisa da máquina tenha acesso real a ela.
