ABI nativa

Interoperabilidade C via OM

Bibliotecas C entram como ABI nativa no LLVM. O usuário escreve Snask; o OM (Orquestrador de Memória) tenta deduzir contratos de recursos a partir dos headers automaticamente, eliminando a necessidade de bindings manuais.

Como Funciona

A interoperabilidade C do Snask não é feita via FFI genérico ou bindings escritos à mão. Em vez disso, o sistema usa o scanner OM para ler headers C diretamente com Clang, extrair funções, tipos, enums e macros, e produzir um contrato que o compilador Snask usa para gerar chamadas LLVM nativas.

O resultado é que você escreve Snask puro — sem arquivos .c, sem bindings manuais, sem arquivos de configuração de FFI. O sistema se encarrega de:

  • Parsear o header C com Clang (análise sintática e semântica completa);
  • Extrair funções, tipos, enums, macros e constantes;
  • Aplicar heurísticas de ownership (construtor, destrutor, método, função pura);
  • Produzir um contrato em memória que o LLVM usa para emitir chamadas nativas;
  • Registrar recursos opacos no runtime OM para limpeza automática via zonas.
Frase guia: você importa um header C como se fosse um módulo Snask. O compilador faz o resto.

Scanner OM

O scanner OM é a ferramenta que analisa headers C e produz contratos. Use-o manualmente para diagnóstico ou deixe o compilador chamá-lo automaticamente durante o build.

Uso básico

snask om scan SDL2/SDL.h --lib sdl2 --output /tmp/sdl2.generated.om.snif

Parâmetros:

ParâmetroDescrição
snask om scan <header>Caminho do header C a ser analisado
--lib <nome>Nome da biblioteca (usado como namespace)
--output <arquivo>Opcional: exporta o contrato para arquivo .om.snif
--patches <dir>Diretório com patches .om.snif para aplicar sobre o contrato gerado

O contrato gerado mostra:

  • Constantes extraídas (macros numéricas, enums);
  • Funções mapeadas com seus níveis de segurança;
  • Recursos descobertos (pares construtor/destrutor);
  • Funções bloqueadas (que o scanner considerou inseguras);
  • Razões para cada classificação.
Dica: sempre gere o contrato com --output na primeira vez que importar uma biblioteca nova. Revise as funções bloqueadas e decida se você precisa de um patch .om.snif.

Sintaxe de Import

A importação de uma biblioteca C em Snask usa a sintaxe import_c_om: 

import_c_om "SDL2/SDL.h" as sdl2

class main {
    fun start() {
        zone "sdl" {
            sdl2.init(sdl2.INIT_VIDEO)
            let window = sdl2.create_window("Hello", 100, 100, 800, 600, 0)
            let renderer = sdl2.create_renderer(window, -1, 0)
            sdl2.render_clear(renderer)
            sdl2.render_present(renderer)
            sdl2.delay(2000)
        }
    }
}

A sintaxe completa do import é:

import_c_om "caminho/para/header.h" as namespace
import_c_om "SDL2/SDL.h" as sdl2        // importa e usa como sdl2.funcao()
import_c_om "GL/gl.h" as gl              // OpenGL
import_c_om "zlib.h" as zlib             // zlib
import_c_om "stdio.h" as stdio           // stdio (puts, printf, etc.)

O caminho do header pode ser:

  • Absoluto ou relativo ao arquivo fonte;
  • Resolvido via -I flags do compilador;
  • Resolvido via pkg-config (veja seção abaixo).

Modelo em 4 Passos

O pipeline de interop C do Snask segue exatamente 4 passos:

PassoDescriçãoFerramenta
1. Scanner lê o header com Clang O scanner invoca o Clang para fazer o parse completo do header C. Extrai funções, tipos, enums, macros, structs e typedefs. snask om scan
2. Heurísticas classificam O scanner aplica heurísticas para classificar cada função em: construtor, destrutor, método, função segura, cópia ou bloqueada. A classificação é conservadora. Scanner OM (automático)
3. Contrato final entra no compilador O contrato gerado (possivelmente com patches .om.snif aplicados) é carregado pelo compilador Snask. Cada função vira um símbolo com assinatura e nível de segurança. Compilador Snask
4. LLVM emite chamada nativa O codegen LLVM emite uma chamada de função direta para a biblioteca C. Recursos opacos são registrados no runtime OM para cleanup automático via zonas. LLVM + runtime OM
Importante: o passo 2 é o mais crítico. Se a heurística classificar uma função perigosa como segura, o usuário pode correr riscos. Se classificar uma função segura como bloqueada, o usuário precisa de um patch. O scanner prefere o erro conservador (bloquear demais) a expor uma função insegura.

O que Funciona Hoje

O scanner OM cobre a maioria dos casos comuns de chamada de bibliotecas C:

CategoriaFuncionaExemplo
Parâmetros numéricossimsdl2.delay(50)
Parâmetros stringsimsdl2.create_window("title", ...)
Retorno numéricosimsdl2.get_ticks()
Retorno voidsimsdl2.render_clear(renderer)
const char* (copiável)simsdl2.get_platform() (retorna string Snask)
Recursos opacos (construtor/destrutor)simsdl2.create_window / SDL_DestroyWindow
Métodos com resource como primeiro parâmetrosimsdl2.render_clear(renderer)
Linking via pkg-configsimpkg-config sdl2 --cflags --libs
Override via .om.snifsimPatch de funções bloqueadas ou mal classificadas

Como o scanner identifica recursos opacos

O scanner busca pares que seguem convenções C comuns:

  • Função que retorna Tipo* → candidato a construtor;
  • Função void destroy(Tipo*) ou void free(Tipo*) → candidato a destrutor;
  • Função que recebe Tipo* como primeiro argumento → candidato a método do recurso;
  • Prefixos comuns: Create, Destroy, New, Free, Open, Close.

Se o scanner encontrar um par, o recurso é registrado no runtime OM. Quando a zona fecha, o destrutor é chamado automaticamente. O usuário nunca vê o ponteiro C cru.

Limitações Atuais

O scanner é conservador por design e ainda não cobre todos os padrões C. Estas são as limitações conhecidas:

CategoriaStatusDescrição
Callbacks limitado Function pointers como parâmetro ainda não são mapeados. APIs que registram callbacks (ex: SDL_AddEventWatch) são bloqueadas.
Function pointers complexos limitado Retorno ou campo de struct com function pointer não é suportado.
Variadic avançado limitado printf com número variável de argumentos não é mapeado automaticamente. Funções com ... são bloqueadas.
Structs por valor limitado Structs C passadas ou retornadas por valor ainda não têm suporte. Apenas ponteiros opacos (Tipo*) funcionam.
Arrays e buffers mutáveis limitado Parâmetros char* ou void* com semântica de buffer mutável não são distinguidos de strings de entrada.
Output pointers com ownership ambíguo limitado Parâmetros de saída (Tipo**) onde o caller pode ser responsável por liberar são bloqueados.
Ponteiros globais limitado Variáveis globais C (ex: extern int*) não são expostas.
Destrutores dependentes de contexto limitado Quando o destrutor correto depende do estado do recurso, o scanner não consegue determinar automaticamente.
Thread affinity limitado Recursos que só podem ser destruídos na thread que os criou não são detectados.
Unions limitado Unions C não são mapeadas para tipos Snask.
Bitfields limitado Campos bitfield em structs não são suportados.
Nota: essas limitações são conhecidas e estão no roadmap. Para todas elas, a saída atual é o patch .om.snif ou um wrapper manual em Systems com @unsafe.

Níveis de Segurança

Cada função no contrato gerado recebe um nível de segurança. Esses níveis determinam como a função é exposta na superfície Snask:

NívelSignificadoExemplo
SAFE Pode ser chamada diretamente da superfície Snask. O scanner verificou que todos os parâmetros e retorno são seguros. sdl2.delay(50)
sdl2.init(sdl2.INIT_VIDEO)
COPY_ONLY Pode ser usada, mas o runtime OM copia o resultado para memória Snask. Usado para funções que retornam const char* onde o ownership não é transferido. sdl2.get_platform() — copia string
BLOCKED Não é exposta na superfície Snask. A função é considerada perigosa ou não mapeável pelo scanner. Requer patch .om.snif para ser liberada. Destrutores, callbacks, funções com void*

Critérios para SAFE

Uma função é classificada como SAFE quando:

  • Todos os parâmetros são tipos numéricos, strings, ou recursos opacos já registrados;
  • O retorno é void, numérico, string copiável, ou recurso opaco com destrutor conhecido;
  • Não há ponteiros de saída com ownership ambíguo;
  • Não há function pointers nos parâmetros.

Critérios para BLOCKED

Uma função é sempre bloqueada quando:

  • É identificada como destrutor (para evitar double-free);
  • Recebe void* sem contexto de tamanho;
  • Usa ... (variadic) — o scanner bloqueia, mas .om.snif com variadic: true desbloqueia;
  • Retorna void*;
  • Envolve function pointers.
Regra de ouro: o scanner prefere falso positivo (bloquear função segura) a falso negativo (expor função perigosa). Patches .om.snif existem para corrigir os falsos positivos.

Sistema de Patch .om.snif

O arquivo .om.snif é um patch opcional sobre o contrato deduzido. Ele não substitui o scanner — apenas corrige casos específicos. Uma biblioteca enorme pode ter um patch de 5 linhas.

Quando usar patch

  • Renomear uma função mal mapeada;
  • Adicionar safety: BLOCKED a uma função perigosa que o scanner classificou como SAFE;
  • Declarar um recurso manualmente (construtor/destrutor não pareado);
  • Corrigir ownership ambíguo com COPY_ONLY;
  • Documentar decisões de segurança.

Exemplo de patch

// contracts/sdl2.om.snif
library sdl2

// Função segura bloqueada pelo scanner conservador
function get_performance_counter:
    c_function: SDL_GetPerformanceCounter
    surface: sdl2.performance_counter
    input: none
    output: value
    safety: SAFE
    reason: returns Uint64, safe numeric type

// Função com string const que precisa de cópia
function get_error:
    c_function: SDL_GetError
    surface: sdl2.error
    input: none
    output: copy
    safety: COPY_ONLY
    reason: returns const char*, must be copied to Snask string
Atenção: o patch não é um binding completo. Ele é aplicado sobre o contrato gerado. Tudo que não estiver no patch continua vindo do scanner.

Exemplo Completo: SDL2

Abaixo, um exemplo funcional de uso do SDL2 via OM. O exemplo inicializa o SDL, cria uma janela com renderer, pinta a tela de preto, espera 2 segundos e finaliza — tudo com gerenciamento automático via zonas. 

import_c_om "SDL2/SDL.h" as sdl2

class main {
    fun start() {
        zone "sdl_lifecycle" {
            // Inicializa SDL com subsistema de vídeo
            if sdl2.init(sdl2.INIT_VIDEO) != 0 {
                print("Falha ao inicializar SDL2")
                return
            }

            // Cria janela — recurso registrado no OM
            let window = sdl2.create_window(
                "Snask + SDL2",
                100,    // x
                100,    // y
                800,    // largura
                600,    // altura
                sdl2.WINDOW_HIDDEN  // flags
            )

            // Cria renderer — também registrado
            let renderer = sdl2.create_renderer(
                window,
                -1,    // driver index (default)
                0      // flags
            )

            // Mostra a janela
            sdl2.show_window(window)

            // Loop de renderização simples
            zone "frame" {
                sdl2.set_render_draw_color(renderer, 0, 0, 0, 255)
                sdl2.render_clear(renderer)
                sdl2.render_present(renderer)
            }

            // Aguarda 2 segundos
            sdl2.delay(2000)

        } // ← window e renderer são destruídos automaticamente aqui
          //   SDL_DestroyRenderer e SDL_DestroyWindow chamados pelo OM
    }
}

O que acontece nos bastidores

  1. sdl2.init é chamada como função SAFE direta;
  2. sdl2.create_window retorna um recurso opaco. O runtime OM registra o ponteiro e associa SDL_DestroyWindow como destrutor;
  3. sdl2.create_renderer recebe o recurso window como primeiro parâmetro (método) e retorna outro recurso;
  4. As funções render_clear, render_present recebem o recurso renderer como primeiro parâmetro;
  5. Ao fechar a zona "sdl_lifecycle", o OM chama SDL_DestroyRenderer e SDL_DestroyWindow na ordem inversa;
  6. O programador nunca viu um SDL_Window* ou chamou free.
Dica: zonas aninhadas permitem ciclos de vida mais granulares. No exemplo, a zona "frame" poderia conter alocações temporárias de textura que são liberadas a cada quadro.

Linking com pkg-config

O Snask pode usar pkg-config para localizar headers e bibliotecas automaticamente. Isso elimina a necessidade de configurar caminhos manualmente.

Como funciona

Quando você usa import_c_om "SDL2/SDL.h" as sdl2, o compilador procura por sdl2 no pkg-config. Se encontrado, ele usa as flags de compilação e linking automaticamente: 

# O compilador faz internamente:
pkg-config sdl2 --cflags   # → -I/usr/include/SDL2
pkg-config sdl2 --libs     # → -lSDL2

Para bibliotecas com nome diferente do namespace, especifique manualmente: 

// Força o nome do pacote pkg-config
import_c_om "SDL2/SDL.h" as sdl2 using "sdl2"

Resolução de caminhos

A ordem de resolução do header é:

  1. Caminho absoluto ou relativo fornecido no import_c_om;
  2. Flags -I passadas ao compilador;
  3. pkg-config --cflags para o nome da biblioteca;
  4. Diretórios padrão do sistema (/usr/include, /usr/local/include).
Nota: pkg-config precisa estar instalado e o pacote correspondente disponível. Para sistemas sem pkg-config, passe os caminhos manualmente com flags do compilador.

Boas Práticas para Interoperabilidade C

Sempre inspecione o contrato

Use snask om scan para gerar e revisar o contrato antes de compilar. Verifique funções bloqueadas e níveis de segurança.

Use zonas para recursos C

Nunca gerencie recursos C manualmente. Coloque toda chamada que cria um recurso dentro de uma zona. O OM limpa automaticamente.

Patches pequenos e focados

Um patch .om.snif com 5-10 regras é saudável. Mais que 50 linhas sugere que o scanner precisa de melhoria, não mais patches.

Evite expor ponteiros

Nunca use @unsafe para expor ponteiros C crus para código seguro. Prefira COPY_ONLY para strings e recursos opacos para objetos.

Diagnostique com trace

Use SNASK_OM_TRACE=1 para ver o ciclo de vida dos recursos: registro, uso e destruição. Essencial para debugar contratos.

Teste com leak check

Use snask test --leak-check para detectar vazamentos de recursos C. O runtime OM instrumenta todas as alocações em modo debug.

Resumo do fluxo de trabalho

  1. Importe o header com import_c_om "minhalib.h" as lib;
  2. Gere o contrato com snask om scan e revise funções bloqueadas;
  3. Crie patches .om.snif apenas para as exceções necessárias;
  4. Escreva código Snask usando as funções expostas dentro de zonas;
  5. Teste com leak detection e trace ativado.

Bibliotecas C entram como ABI nativa no LLVM. O usuário escreve Snask; o OM tenta deduzir contratos de recursos a partir dos headers. O resultado é uma chamada nativa, sem overhead, com gerenciamento automático de ciclo de vida.
