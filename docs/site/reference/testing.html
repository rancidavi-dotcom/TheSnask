Testes da Documentacao — Confiabilidade

Exemplos de documentacao so contam como teste real quando existem como arquivo .snask versionado no repositorio e passam por snask build no CI. Esta pagina documenta a politica, o fluxo de trabalho e como verificar a cobertura.

# Politica de honestidade

A documentacao da Snask segue uma politica rigorosa de honestidade tecnica:

  • Se compila e linka hoje, a pagina pode apontar para um teste real versionado.
  • Se existe apenas no analisador semantico, a pagina fica marcada sem teste versionado.
  • Se depende de GUI ou ambiente externo, o exemplo deve explicar a dependencia antes do codigo.
  • Se a feature e parcial ou experimental, a pagina deve dizer isso antes de mostrar o exemplo.
Nunca copie um exemplo da documentacao esperando que ele compile se a pagina avisa que a feature e experimental. Leia o Status Real antes de confiar em um exemplo.

# Como executar todos os testes de documentacao

Use o script automatizado que varre todos os exemplos e tenta compilar cada um:

scripts/check_doc_examples.sh

O script retorna um codigo de saida zero se todos os exemplos compilarem, ou lista os arquivos que falharam com o diagnostico do compilador. Para um exemplo especifico:

snask build docs/examples/reference/basics/hello.snask

Executar um grupo especifico

scripts/check_doc_examples.sh --group=collections
scripts/check_doc_examples.sh --group=systems

Adicionar um novo exemplo ao conjunto de teste

  1. Crie o arquivo .snask em docs/examples/reference/<categoria>/.
  2. Verifique se compila com snask build.
  3. Adicione uma entrada na pagina HTML da funcao correspondente.
  4. O CI vai executa-lo automaticamente no proximo push.

Cada pagina de funcao que possui exemplo validado mostra o comando exato de compilacao no topo do exemplo. Por exemplo, na pagina len:

# compilar: snask build docs/examples/reference/collections/len.snask

# Exemplo completo: pagina de funcao com teste

Abaixo, o formato padrao que toda pagina de funcao deve seguir quando possui um teste versionado:

<section id="exemplo">
  <h2>Exemplo&lt;/h2>
  <p>Este exemplo faz parte do conjunto de testes da documentacao.</p>
  <pre><code>class main {
    fun start() {
        let x = 42
        print("{x}\n")
    }
}</code></pre>
  <p>Compile com:</p>
  <pre><code>snask build docs/examples/reference/basics/print-example.snask</code></pre>
</section>
Toda pagina de funcao em docs/site/reference/functions/ deve seguir este formato quando a funcao e estavel. Se a funcao e experimental, o paragrafo inicial deve dizer "Esta funcao e experimental — pode mudar sem aviso."

# Integracao continua

O repositorio executa scripts/check_doc_examples.sh em todo PR e push para a branch principal. Se um exemplo quebrar, o CI falha e o autor e notificado. Exemplos que dependem de GUI ou de hardware especifico sao marcados com # skip-ci no cabecalho do arquivo e nao bloqueiam o pipeline.

TriggerAcao
Push para mainExecuta suite completa de exemplos + testes unitarios
Pull requestExecuta suite completa de exemplos
Diario (cron)Executa suite completa + exemplos skip-ci (se ambiente disponivel)
